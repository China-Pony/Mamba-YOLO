## B-2：单独增大 P2 基础通道数

### 文件：`ultralytics/cfg/models/mamba-yolo/Mamba-YOLO-T-C2.yaml`

P2 的 128 在 Tiny `width=0.25` 下变为 32ch。改为 256 变为 64ch。

**原始 backbone（第 1-2 行）：**
```yaml
  - [-1, 1, SimpleStem, [128, 3]]   # 0-P2/4  → 缩放后 32ch
  - [-1, 3, VSSBlock, [128]]        # 1       → 缩放后 32ch
```

**修改为：**
```yaml
  - [-1, 1, SimpleStem, [256, 3]]   # 0-P2/4  → 缩放后 64ch
  - [-1, 6, VSSBlock, [256]]        # 1       → 缩放后 64ch
```

> ⚠️ 这会显著增加参数量。P2 通道增大后，后续 VisionClueMerge 到 P3 的下采样通道也要同步增大：
> `VisionClueMerge(64ch * 4 → 256ch → 缩放后 64ch)`，但原来 P3 基础就是 256。所以这里可能需要调整 `VisionClueMerge` 的 out_dim 参数。

完整调整后的 backbone：
```yaml
backbone:
  - [-1, 1, SimpleStem, [256, 3]]   # 0-P2/4, C=64 (width=0.25)
  - [-1, 6, VSSBlock, [256]]        # 1
  - [-1, 1, VisionClueMerge, [256]] # 2-P3/8  (原 256 不变)
  - [-1, 3, VSSBlock, [256]]        # 3
  - [-1, 1, VisionClueMerge, [512]] # 4-P4/16
  - [-1, 9, VSSBlock, [512]]        # 5
  - [-1, 1, VisionClueMerge, [1024]]# 6-P5/32
  - [-1, 3, VSSBlock, [1024]]       # 7
  - [-1, 1, SPPF, [1024, 5]]        # 8
```

对应的 neck（Concated 时 index 不变，但通道数自动适配，不需要额外修改）。

## B-3：用 VisionClueMerge 替换 neck 中的普通 Conv 下采样

在 C2 的 neck 中，从 P2→P3→P4→P5 的下采样用的是 `Conv` 而非 `VisionClueMerge`。

### 文件：`ultralytics/cfg/models/mamba-yolo/Mamba-YOLO-T-C2.yaml`

**原始 neck（第 29-39 行）：**
```yaml
  - [-1, 1, Conv, [128, 3, 2]]     # P2→P3 下采样
  - [[-1, 14], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [256]]

  - [-1, 1, Conv, [256, 3, 2]]     # P3→P4 下采样
  - [[-1, 11], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [512]]

  - [-1, 1, Conv, [512, 3, 2]]     # P4→P5 下采样
  - [[-1, 8], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [1024]]
```

**修改为（替换 Conv 为 VisionClueMerge）：**
```yaml
  - [-1, 1, VisionClueMerge, [128]]     # P2→P3 下采样（4倍通道合并再压缩）
  - [[-1, 14], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [256]]

  - [-1, 1, VisionClueMerge, [256]]     # P3→P4 下采样
  - [[-1, 11], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [512]]

  - [-1, 1, VisionClueMerge, [512]]     # P4→P5 下采样
  - [[-1, 8], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [1024]]
```

> ⚠️ VisionClueMerge 是像素重组下采样（4倍通道→1x1压缩），输入通道=输出通道/4 即可。这里 Conv 的 args 是 `[output_ch, kernel, stride]`，需改为 VisionClueMerge 的 `[output_ch]`。

实际上 VisionClueMerge 会把输入拆成4个子图再合并，输入通道应为 `output_ch * 4`。但由于 YAML 中 `c1` 会从之前层自动推断，所以只需要设置 `output_ch`。

但更简单的方式是：**只改 P2→P3 这一级下采样**（最浅层信息最珍贵），P3→P4 和 P4→P5 保留 Conv 防止参数量过大：

```yaml
  - [-1, 1, VisionClueMerge, [128]]     # 只改 P2→P3 为 VisionClueMerge
  - [[-1, 14], 1, Concat, [1]]
  - [-1, 3, XSSBlock, [256]]

  - [-1, 1, Conv, [256, 3, 2]]          # P3→P4 保留 Conv
  ...
```

### ⚠️ VisionClueMerge 通道注意事项

`VisionClueMerge` 类（[mamba_yolo.py:L477-L495](file:///d:/Desktop/Mamba-YOLO/ultralytics/nn/modules/mamba_yolo.py#L477-L495)）内部实现：
```python
self.hidden = int(dim * 4)   # dim = 输入通道
self.pw_linear = Conv2d(self.hidden, out_dim, 1)
```

所以实际流程是：输入 C → 4倍通道拼接 → 1×1压缩到 out_dim。在 tasks.py 的 parse_model 中，VisionClueMerge 被特殊处理（L901），args[0] 被用作 `c2 = min(out_dim * width, max_channels)`。

当替换 neck 中的 Conv 为 VisionClueMerge 时，需注意：
- 原来 Conv 的 `args = [out_c, kernel, stride]` → `c2 = make_divisible(out_c * width, 8)`
- VisionClueMerge 的 `args = [out_dim]` → 同样的 c2 逻辑

所以 YAML 中写 `VisionClueMerge, [128]` 即可（缩放后 32ch）。
