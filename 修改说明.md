## 方案2：修复数据预处理一致性

### 修改文件2：`ultralytics/models/mamba_yolo_rtdetr/val.py`（第21行）

**问题：** 训练时 `stretch=True` 导致图像被直接拉伸（失去宽高比），这与验证时使用的LetterBox预处理不一致。

```python
# ==== 第18-21行 修改前 ====
def build_transforms(self, hyp=None):
    if self.augment:
        hyp.mosaic = hyp.mosaic if self.augment and not self.rect else 0.0
        hyp.mixup = hyp.mixup if self.augment and not self.rect else 0.0
        transforms = v8_transforms(self, self.imgsz, hyp, stretch=True)

# ==== 修改后 ====
def build_transforms(self, hyp=None):
    if self.augment:
        hyp.mosaic = hyp.mosaic if self.augment and not self.rect else 0.0
        hyp.mixup = hyp.mixup if self.augment and not self.rect else 0.0
        transforms = v8_transforms(self, self.imgsz, hyp, stretch=False)
```

**为什么这样改？**

| 参数        | stretch=True             | stretch=False           |
| --------- | ------------------------ | ----------------------- |
| 行为        | 图像直接拉伸填充640x640          | LetterBox保持宽高比，额外区域填充灰边 |
| 宽高比       | ❌ 被破坏                    | ✅ 保持                    |
| 与验证一致性    | ❌ 不一致                    | ✅ 一致                    |
| 对RTDETR影响 | 特征几何关系扭曲，Transformer难以学习 | 一致的几何关系                 |

**额外影响分析：**
`stretch=True` 时 `RandomPerspective` 的 `LetterBox(new_shape=(imgsz, imgsz))` 被设为 `None`，这意味着：

1. Mosaic拼接后的图像直接resize到640x640
2. 目标的宽高比在拼接+变形双重影响下严重失真
3. 矩形目标可能变为正方形，长条形目标被压扁

对于RT-DETR的Transformer Decoder头，它需要从特征中学习目标的几何先验（通过deformable attention），扭曲的几何关系会使这个学习过程更加困难。

## 方案3：调整Mamba-YOLO-T-rtdetr YAML中的RTDETRDecoder配置

### 修改文件3：`ultralytics/cfg/models/mamba-yolo/Mamba-YOLO-T-rtdetr.yaml`

**分析：** Mamba-YOLO-T的width=0.25，PAFPN输出通道实际值约为(64, 128, 256)。RTDETRDecoder默认期望(512, 1024, 2048)的通道输入。虽然 `input_proj` 用1x1 Conv进行了投影，但过小的特征通道意味着传入Transformer decoder的信息量不足。

**修改方案：**

```yaml
# ==== 原始配置 ====
head:
  - [[14, 17, 20], 1, RTDETRDecoder, [nc]]

# ==== 方案A：调整Decoder内部参数（推荐，效果最好）====
head:
  - [[14, 17, 20], 1, RTDETRDecoder, [nc, 256, 100, 4, 4, 3, 512]]

# 对应参数解释：
#   nc=80         # 类别数
#   hd=256        # hidden dim（保持默认）
#   nq=100        # queries数量从300减少到100，降低匹配难度
#   ndp=4         # decoder points数（保持默认）
#   nh=4          # attention heads从8减少到4（因为特征通道小）
#   ndl=3         # decoder layers从6减少到3，简化训练
#   d_ffn=512     # FFN维度从1024减少到512（与特征容量匹配）

# ==== 方案B：保守调整（如果方案A效果过度，试试这个）====
head:
  - [[14, 17, 20], 1, RTDETRDecoder, [nc, 256, 150, 4, 6, 4, 768]]

# 对应参数：
#   nc=80, hd=256, nq=150, ndp=4, nh=6, ndl=4, d_ffn=768
```

**参数调优理由详解：**

| 参数                     | 默认值  | 建议值     | 原因                                             |
| ---------------------- | ---- | ------- | ---------------------------------------------- |
| `nq` (num queries)     | 300  | 100-150 | COCO子集目标更少，300 queries冗余度高，增加匹配难度。减少到150可加快收敛  |
| `ndl` (decoder layers) | 6    | 3-4     | 6层decoder需要更多数据训练，100 epochs不足以充分训练。减到3层使训练更高效 |
| `nh` (num heads)       | 8    | 4-6     | 输入特征通道只有256，8头注意力每个头只分到32维，不足以学习有区分力的特征        |
| `d_ffn`                | 1024 | 512-768 | FFN的容量需要与输入特征容量匹配，太大的FFN在数据量少时容易过拟合            |

## 方案4：增强input\_proj特征适配层

### 修改文件4：`ultralytics/nn/modules/head.py`

**问题：** 默认的 `input_proj` 只是 `Conv2d + BN`，线性表达能力有限，不能有效弥合SSM特征和CNN特征之间的分布差异。

```python
# ==== 第320行附近 修改前 ====
# Backbone feature projection
self.input_proj = nn.ModuleList(
    nn.Sequential(nn.Conv2d(x, hd, 1, bias=False), nn.BatchNorm2d(hd)) for x in ch
)

# ==== 修改后：增加非线性变换层 ====
# Backbone feature projection (enhanced for SSM features)
self.input_proj = nn.ModuleList(
    nn.Sequential(
        nn.Conv2d(x, hd, 1, bias=False),
        nn.BatchNorm2d(hd),
        nn.Conv2d(hd, hd, 3, padding=1, bias=False),  # 增加3x3卷积提取空间特征
        nn.BatchNorm2d(hd),
        nn.ReLU(inplace=True),
    ) for x in ch
)
```

**设计思路：**

- 第一层1x1 Conv：通道投影（与原来相同）
- 第二层3x3 Conv + BN + ReLU：引入空间信息提取能力
  - SSM（Mamba-YOLO）的特征更关注序列性/全局依赖
  - 3x3卷积的特征更关注局部纹理/边缘
  - 通过3x3卷积补充局部特征，使Decoder更容易检测目标
- 保留BN：归一化特征分布，消除SSM特征和CNN特征的尺度差异

**替代方案（更轻量的改进）：**

```python
# 仅增加一个GELU激活层
self.input_proj = nn.ModuleList(
    nn.Sequential(
        nn.Conv2d(x, hd, 1, bias=False), 
        nn.BatchNorm2d(hd),
        nn.GELU()  # GELU比ReLU更平滑，适合Transformer架构
    ) for x in ch
)
```

## 方案5：添加LayerNorm预归一化

### 修改文件5：创建新的适配模块

**背景：** Mamba-YOLO backbone的SS2D模块输出特征与RTDETRDecoder期望的特征分布可能不同。VSSBlock内部使用了LayerNorm，但PAFPN的输出没有做归一化。

**新文件建议：** 在 `RTDETRDecoder` 中增加可选的特征预归一化。

```python
# 在 RTDETRDecoder.__init__ 中增加（head.py L320附近）：
# 在 input_proj 之后增加
self.prenorm = nn.ModuleList(
    nn.LayerNorm(hd) for _ in ch  # 对每个尺度的特征分别做LayerNorm
)

# 在 _get_encoder_input 中（head.py L410附近）：
def _get_encoder_input(self, x):
    # Get projection features
    x = [self.input_proj[i](feat) for i, feat in enumerate(x)]
    # ==== 添加：特征预归一化 ====
    x = [self.prenorm[i](feat.permute(0, 2, 3, 1)).permute(0, 3, 1, 2) for i, feat in enumerate(x)]
    # ==== 结束添加 ====
    # Get encoder inputs
    feats = []
    shapes = []
    for feat in x:
        h, w = feat.shape[2:]
        feats.append(feat.flatten(2).permute(0, 2, 1))
        shapes.append([h, w])
    feats = torch.cat(feats, 1)
    return feats, shapes
```

**为什么需要LayerNorm：**

- RTDETR原本用ResNet backbone，其输出特征经过多层BN归一化，数值稳定
- Mamba-YOLO的SS2D使用LayerNorm（非BN），特征统计特性不同
- 额外的LayerNorm可以统一特征分布，降低Transformer Encoder的学习难度

## 方案7：两阶段训练策略（高级方案）

这个方案需要创建一个新的训练脚本，先冻结backbone训练decoder，再联合微调。

### 新文件：`mbyolo_rtdetr_train_stage2.py`

```python
"""
Mamba-YOLO + RTDETR Head 两阶段训练策略

阶段1：冻结backbone，只训练RTDETRDecoder头（20 epochs）
阶段2：联合微调整个模型（100+ epochs）

适用场景：当从头训练时，RTDETRDecoder的随机初始化导致收敛困难
"""

from ultralytics.models.mamba_yolo_rtdetr import MambaYOLORTDETR
import argparse
import os
import torch

ROOT = os.path.abspath('.') + "/"

def parse_opt():
    parser = argparse.ArgumentParser()
    parser.add_argument('--data', type=str, default=ROOT + '/ultralytics/cfg/datasets/coco.yaml')
    parser.add_argument('--config', type=str, default=ROOT + '/ultralytics/cfg/models/mamba-yolo/Mamba-YOLO-T-rtdetr.yaml')
    parser.add_argument('--batch_size', type=int, default=32)
    parser.add_argument('--imgsz', '--img', '--img-size', type=int, default=640)
    parser.add_argument('--device', default='0')
    parser.add_argument('--workers', type=int, default=8)
    parser.add_argument('--epochs', type=int, default=150)
    parser.add_argument('--project', default=ROOT + '/output_dir/mscoco_rtdetr')
    parser.add_argument('--name', default='mambayolo_rtdetr_v2')
    opt = parser.parse_args()
    return opt

if __name__ == '__main__':
    opt = parse_opt()
    
    # ====== 阶段1：只训练Decoder头（冻结backbone）====== 
    print("=" * 60)
    print("阶段1：冻结backbone，只训练RTDETRDecoder头")
    print("=" * 60)
    
    model = MambaYOLORTDETR(ROOT + opt.config)
    
    # 冻结所有backbone参数（模型中除最后一个模块外的所有层）
    for name, param in model.model.named_parameters():
        if 'model.-1' not in name:  # -1 是最后一层（RTDETRDecoder）
            param.requires_grad = False
    
    # 冻结训练20个epochs，使用较高学习率
    model.train(
        data=ROOT + opt.data,
        epochs=20,
        workers=opt.workers,
        batch=opt.batch_size,
        optimizer="AdamW",
        lr0=0.001,                 # 训练头用较高学习率
        lrf=0.1,
        weight_decay=0.0001,
        warmup_epochs=1.0,
        warmup_bias_lr=0.01,
        close_mosaic=0,
        device=opt.device,
        amp=True,
        project=ROOT + opt.project,
        name=opt.name + "_stage1",
        exist_ok=True,
    )
    
    # 保存阶段1的权重
    stage1_weights = ROOT + opt.project + f"/{opt.name}_stage1/weights/best.pt"
    
    # ====== 阶段2：联合微调 ======
    print("=" * 60)
    print("阶段2：加载阶段1权重，联合微调整个模型")
    print("=" * 60)
    
    # 重新加载模型（恢复所有参数为可训练）
    model = MambaYOLORTDETR(ROOT + opt.config)
    
    # 加载阶段1的最佳权重（decoder头已经初始化好）
    if os.path.exists(stage1_weights):
        model.load(stage1_weights)
    
    # 联合微调
    model.train(
        data=ROOT + opt.data,
        epochs=opt.epochs,
        workers=opt.workers,
        batch=opt.batch_size,
```

## 方案8：排查AMP兼容性并启用

### 修改文件：`mbyolo_rtdetr_train.py`

**背景：** 基线使用了AMP（`amp=True`），但RTDETR训练脚本默认关闭了AMP，并标注了"CAUTION: may cause NaN with RT-DETR"。

如果直接启用AMP出现问题（NaN loss），可以添加梯度裁剪和GradScaler配置：

```python
# 在 mbyolo_rtdetr_train.py 的 args 中添加：
args = {
    # ... 其他参数
    "amp": True,
    # 以下参数通过 model.train() 传递给训练引擎
    # gradient_clip_val: 1.0,  # 梯度裁剪（需要查看trainer.py是否支持）
}
```

如果框架不支持直接传入梯度裁剪参数，你需要在 `ultralytics/engine/trainer.py` 中修改：

```python
# ==== 在 trainer.py 中找到loss backward部分 ====
# 修改前
self.scaler.scale(loss).backward()

# 修改后（添加梯度裁剪）
self.scaler.scale(loss).backward()
if self.args.get('grad_clip_val', 0) > 0:
    self.scaler.unscale_(self.optimizer)
    torch.nn.utils.clip_grad_norm_(self.model.parameters(), self.args['grad_clip_val'])
```

## 预期提升

| 指标       | 当前值   | 预期值（优化后）      | 优化方向                         |
| -------- | ----- | ------------- | ---------------------------- |
| mAP50    | 0.172 | **0.25-0.28** | 训练配置+数据增强修正                  |
| mAP50-95 | 0.107 | **0.16-0.18** | 同上 + Decoder配置优化             |
| Recall   | 0.211 | **0.28-0.30** | close\_mosaic + 更合理的queries数 |

**为什么无法达到基线0.313 mAP50的性能？**
因为RT-DETR本身就是一个**端到端架构**，它不使用NMS、不需要调节置信度阈值、不需要DFL损失。但这也意味着它需要**更多的数据**和**更长的训练时间**来充分学习。在COCO子集+100 epochs的限制下，mAP低于原版YOLO head是预期之内的事情。

如果希望RTDETR Head真正超越原版YOLO Head，建议：

1. 使用完整的COCO数据集
2. 训练至少300 epochs（RT-DETR官方是6x schedule ≈ 72 epochs on full COCO，但你的模型更大）
3. 使用Mamba-YOLO-B或Mamba-YOLO-L backbone，提供更丰富的特征
